SLIDE 5 — SECURITY ARCHITECTURE
ESTIMATED TIME: 2–2.5 minutes (this is your CY core slide — go deeper)
---------------------------------------------------------------
[Say this out loud]
"This is the cybersecurity heart of our project. Let me go through each
layer.

First, password hashing. We use Werkzeug's generate_password_hash, which
applies PBKDF2 with SHA-256 and a random salt by default. The raw password
is never stored — not even temporarily. Only the hash goes into the DB.

Second, session-based authentication. Flask-Login handles this. Once logged
in, the user's ID is stored in a server-side session. Every protected route
has the @login_required decorator. If a logged-out user tries to access
the dashboard, they're redirected to login.

Third, CSRF protection. Every POST form in our app is a Flask-WTF FlaskForm.
The template calls form.hidden_tag(), which injects a hidden CSRF token.
If someone tries to forge a POST request from another origin, it fails
validation before any DB operation happens.

Fourth, AES-256-GCM file encryption. When a doctor uploads a report file,
we generate a fresh 256-bit AES key, create a random 12-byte nonce, and
encrypt the file bytes using the AESGCM cipher. The output is the nonce
prepended to the ciphertext, stored as a .enc file on disk.

Fifth, RBAC. Every route starts with a role check. If the current user's
role doesn't match what the route expects, we call abort(403) immediately,
before any data is touched. For reports specifically, we also verify the
patient's ID matches the appointment owner."

[Tip] Go slow on this slide. The audience is CS faculty — they want detail.
[Tip] If asked "why GCM and not CBC?" — say: "GCM provides both
      encryption AND an integrity authentication tag. CBC encrypts but
      doesn't detect tampering. GCM is the NIST-recommended AEAD mode."
[Tip] Point to CIAAA badges on the right — "we'll map these in slide 8."

SLIDE 6 — ENCRYPTION FLOW (VISUAL)
ESTIMATED TIME: 1.5–2 minutes
---------------------------------------------------------------
[Say this out loud]
"Let me trace exactly what happens when a file gets encrypted and then
retrieved.

Upload path: The doctor submits a report with an attached file. Our code
calls generate_key(), which uses AESGCM.generate_key to produce a random
256-bit key and returns it base64-encoded for safe DB storage.

Then encrypt_file() is called. Inside, os.urandom(12) generates a fresh
nonce — 12 bytes of cryptographically random data, unique per file.
AESGCM encrypts the file bytes, producing ciphertext plus a 16-byte
authentication tag built into GCM. We prepend the nonce to the ciphertext
and write everything to disk as a .enc file. The key goes into the database.
The file goes to storage/reports. They're never in the same place.

Download path — the reverse: we read the .enc file from disk, call
decrypt_file() with the base64 key from the DB. The first 12 bytes are
the nonce. The rest is ciphertext. AESGCM.decrypt verifies the GCM tag
and decrypts. The raw bytes go to send_file() in memory — nothing lands
on disk unencrypted.

The callout at the bottom is important: because GCM includes an
authentication tag, if anyone modifies even a single byte of the
ciphertext on disk, decryption will FAIL with an exception. There is no
silent data corruption."

[Tip] Trace the arrows with your hand as you speak. Left to right on upload,
      right to left on download. It helps the audience visualize the flow.
[Tip] If asked "where is the nonce stored?" — "Prepended to the file itself.
      First 12 bytes of every .enc file are the nonce."

SLIDE 7 — EDGE CASES & HANDLING
ESTIMATED TIME: 1–1.5 minutes
---------------------------------------------------------------
[Say this out loud]
"Good security isn't just about the happy path — it's about what happens
when things go wrong. Let me walk through the edge cases we handled.

Invalid login: no information is leaked about whether the username exists

Unauthorized report access: if a patient tries to fetch another patient's
report ID directly from the URL, they hit a 403 immediately — before any
file or data is returned.

Report before a confirmed appointment: the doctor can't create a report
until the appointment is in CONFIRMED status. The route checks this
explicitly and redirects with a warning.

Report already exists: if a doctor somehow resubmits the report form, we
detect the existing report and block the duplicate — no double entries.

No file attached: if the patient tries to download a report that has no
file, they get a clear warning flash instead of a server error.

Appointment status guards: you can only confirm a PENDING appointment,
and you can't cancel something already cancelled. Every mutation is guarded
before it touches the database."

[Tip] Emphasize "no silent failures" — everything surfaces to the user
      as a visible message. This is good UX and good security design.

SLIDE 8 — CIAAA ALIGNMENT
ESTIMATED TIME: 1.5 minutes
---------------------------------------------------------------
[Say this out loud]
"CIAAA is an extension of the classic CIA triad — Confidentiality,
Integrity, Availability — with Authentication and Authorization added
as explicit security pillars. Let me map our project to each one.

Confidentiality: report files are AES-256-GCM encrypted at rest. RBAC
prevents unauthorized users from even reaching the data.

Integrity: GCM's authentication tag means any tampered ciphertext fails
decryption. SQLAlchemy uses parameterized queries, preventing SQL injection
from corrupting the database.

Availability: Flask and SQLite have no external dependencies. The app
starts instantly and doesn't rely on a remote database server or cloud
service.

Authentication: Flask-Login manages sessions. Werkzeug hashes passwords
with PBKDF2. No plaintext passwords anywhere in the system.

Authorization: every route checks the user's role before execution.
Report access additionally checks appointment ownership. Access control
is enforced at the route level, not just the UI level.

The NIST SP 800-53 framework is one authoritative reference for these
principles in real-world security engineering."

[Tip] If asked "what's the difference between Authentication and
      Authorization?" — "Authentication answers WHO are you.
      Authorization answers WHAT are you allowed to do."

        key and ciphertext (different storage) is already a meaningful
        improvement over plaintext storage."
  Q: "Why Flask and not Django?"
     A: "Flask gives us exactly what we need — routing, ORM, auth, forms —
        without a large framework overhead. Lower complexity = fewer places
        for bugs to hide."
  Q: "Is this production-ready?"
     A: "No — and we're transparent about that. It's an academic demo that
        applies production-grade security patterns, which is the point of
        the exercise."
